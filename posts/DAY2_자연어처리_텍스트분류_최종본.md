## 2. 텍스트 분류란? 🟦

**텍스트 분류(Text Classification)** 는 컴퓨터가 문장이나 문서의 내용을 분석해, **미리 정해진 범주(Class, 라벨)** 중 하나 또는 여러 개로 자동 분류하는 자연어 처리 작업입니다.

- **입력(Input)**: 텍스트 데이터(문장·문서)
- **출력(Output)**: 미리 정의된 클래스(Label)

🟦 (강의 예시)

| 입력 문장 | 출력(라벨) |
|---|---|
| 이 스마트폰은 성능이 뛰어나고 사용하기 편리하다. | 긍정 |
| 지금 회원가입하면 최신 노트북을 무료로 드립니다. | 스팸 |

대표적으로 **이메일을 스팸/정상으로 구분**하거나, **영화 리뷰를 긍정/부정으로 분류**하는 일이 텍스트 분류입니다.

### 2.1 왜 필요한가 — 텍스트 분류의 목적 🟦

대량의 문서를 사람이 일일이 읽지 않고 **자동으로 분석**하기 위해 사용합니다. 강의가 든 대표 목적은 다음과 같습니다.

- 문서 자동 분류 · 감성 분석 · 스팸 메일 탐지 · 뉴스 분류
- 고객 의견 분석 · 상품 리뷰 분석 · 의료/법률 문서 분류
- SNS 게시글 분석 · 챗봇 의도(Intent) 분류

### 2.2 전체 과정 한눈에 보기 🟦

일반적인 텍스트 분류는 다음 순서로 진행됩니다.

```text
텍스트 수집
  ↓ 텍스트 전처리 (정제 · 토큰화 · 형태소 분석 · 불용어 제거)
  ↓ 형태소 분석
  ↓ 불용어 제거
  ↓ 벡터화 (BOW · TF-IDF · Word2Vec 등)
  ↓ 모델 학습
  예측 및 평가
```

> 🟩 **핵심 포인트**: 모델은 글자를 직접 이해하지 못합니다. 그래서 **"텍스트 → 숫자(벡터)"** 로 바꾸는 전처리·벡터화 단계가 분류 성능을 크게 좌우합니다. 이 글의 §4가 그 부분입니다.

## 3. 텍스트 분류의 종류 🟦

분류는 **라벨이 몇 개이고, 한 문장이 몇 개의 라벨을 갖는가**에 따라 세 가지로 나뉩니다.

| 종류 | 설명 | 예시 |
|---|---|---|
| **이진 분류**(Binary) | 클래스가 **2개** | 정상 / 비정상, 승인 / 거절 |
| **다중 분류**(Multi-Class) | 클래스가 **3개 이상**, 그중 **하나** 선택 | IT / 교육 / 건강 / 여행 / 음식 |
| **다중 라벨 분류**(Multi-Label) | 한 문장이 **여러 라벨 동시** 보유 | "AI를 활용한 의료 영상 분석…" → `인공지능`, `의료`, `영상처리` |

> 🟩 **주의 — 다중 분류 ≠ 다중 라벨 (모델 출력이 다릅니다)**: 이 둘은 비슷해 보이지만 모델 마지막 층과 손실함수가 다릅니다.
> - **다중 분류**(정답 1개): 출력층은 보통 **softmax**, 모든 클래스 확률의 합이 1이 되도록 만들어 **가장 높은 하나**를 고릅니다. (손실함수는 cross-entropy 계열)
> - **다중 라벨**(정답 여러 개): 라벨마다 **독립적으로 sigmoid** 를 적용해 "이 라벨에 해당하는가(예/아니오)"를 각각 판단합니다. 즉 **라벨마다 0/1 이진 분류**를 따로 하는 셈이라, 손실함수도 `binary_crossentropy` 계열을 라벨 수만큼 적용합니다. softmax를 쓰면 안 됩니다(라벨끼리 확률을 나눠 갖게 되므로).
>
> 입문 단계에서는 대부분 **이진 분류**(스팸/감성)로 시작하면 충분합니다. (이 구분은 강의에 명시되어 있지 않은 🟩 보충 설명입니다.)

## 4. 텍스트를 숫자로 바꾸기: 전처리와 벡터화 🟦

텍스트는 그대로 모델에 넣을 수 없으므로, 여러 단계를 거쳐 숫자로 바꿉니다.

```text
원문: 오늘 새로 출시된 노트북의 성능이 정말 뛰어나다!!
  ↓ 소문자 변환(영문인 경우)
  ↓ 특수문자 제거
  ↓ 형태소 분석
  ↓ 불용어 제거
  ↓ 토큰 생성
  ↓ 벡터화
```

### 4.1 전처리 단계 🟦

| 단계 | 하는 일 | 예시 |
|---|---|---|
| **형태소 분석** | 문장을 의미 있는 최소 단위로 분리 | "학생들이 열심히 공부하고 있다" → `학생`, `열심히`, `공부`, `하다` |
| **불용어 제거** | 의미가 거의 없는 단어 제거 | `학생`, `열심히`, `공부` (조사·어미 등 제거) |
| **토큰화**(Tokenization) | 문장을 단어 단위로 분리 | "서울의 야경은 아름답고 화려하다" → `서울`, `야경`, `아름답다`, `화려하다` |
| **정수 인코딩** | 단어를 숫자 번호로 변환 | `서울→1, 야경→2, 아름답다→3` ⇒ "서울 야경 아름답다" → `1 2 3` |
| **패딩**(Padding) | 문장 길이를 동일하게 맞춤 | `[5 8]` → `[5 8 0 0]` (짧은 문장 뒤를 0으로 채움) |

> 🟩 **토큰화와 형태소 분석은 무엇이 다른가?**: 입문자는 둘을 같은 작업으로 보기 쉽습니다. **토큰화**는 문장을 작은 조각(토큰)으로 "쪼개는 전체 과정"을 가리키고, **형태소 분석**은 그중에서도 한국어처럼 조사·어미가 붙는 언어에서 **의미 단위와 품사를 고려해** 더 언어학적으로 쪼개는 방식입니다. 그래서 한국어에서는 형태소 분석이 곧 좋은 토큰화의 핵심이 됩니다.

> 🟦 **한국어는 형태소 분석이 특히 중요**합니다. 조사("이/가/을/를")와 어미("-하고/-있다")가 많아서, 단순히 띄어쓰기로만 나누면 같은 단어가 제각각 다르게 취급되기 때문입니다. 강의가 소개한 대표 형태소 분석기: **KoNLPy, Mecab, Okt, Komoran, Kkma, Hannanum**.
>
> 🟩 **참고**: 위 분석기들은 대부분 설치 시 **Java(JDK)** 환경이 필요합니다(특히 Windows에서는 JDK·JPype 설치 이슈가 잦습니다). 그래서 이 글의 실습(§4.3, §5)은 형태소 분석 없이도 돌아가는 간단한 예제로 구성했습니다.

🟦 **정수 인코딩과 패딩 예시** (PDF p.8 직접 예시)

토큰을 만든 뒤에는 각 단어에 번호를 붙입니다.

```text
서울 → 1
야경 → 2
아름답다 → 3
화려하다 → 4

"서울 야경 아름답다" → [1, 2, 3]
```

그런데 모델은 여러 문장을 한 번에 묶어 처리하므로, 문장마다 길이가 다르면 곤란합니다. 그래서 짧은 문장 뒤에 `0`을 채워 길이를 맞춥니다.

```text
[5, 8]       → [5, 8, 0, 0]
[2, 4, 7, 9] → [2, 4, 7, 9]
```

> 🟩 여기서 `0`은 실제 단어가 아니라 **"빈 자리"** 를 뜻하는 특별한 값입니다. 이렇게 길이를 맞춰야 여러 문장을 같은 크기의 행렬로 묶어 한 번에 학습할 수 있습니다.

### 4.2 벡터화(Vectorization) 방법들 🟦

컴퓨터가 이해할 수 있도록 토큰을 **숫자 벡터**로 바꾸는 단계입니다. 강의가 소개한 방법들을 **"단어를 세는 방식 → 의미를 담는 방식 → 문맥까지 담는 방식"** 순으로 정리합니다.

| 방법 | 핵심 아이디어 | 특징 |
|---|---|---|
| **BOW**(Bag of Words) | 단어의 **출현 횟수**만 센다 | 간단하지만 단어 **순서·의미**를 무시 |
| **TF-IDF** | 흔한 단어는 낮추고, **특정 문서에만 자주 나오는 단어**는 강조 | 중요한 단어 강조 → 문서 분류 성능 향상 |
| **Word2Vec** | 단어 **의미**를 벡터 공간에 표현 | 의미가 가까우면 벡터도 가까움(자동차↔버스) |
| **FastText** | Word2Vec 개선 | **새로운 단어(미등록어)** 처리 가능, 한국어에 비교적 유리 |
| **GloVe** | 단어의 **전역 통계** 정보 반영 | 말뭉치 전체의 동시출현 정보를 활용 |
| **BERT 임베딩** | **문맥(Context)** 을 고려 | 같은 단어도 문맥 따라 다른 벡터 |

> 🟦 **BERT 임베딩이 특별한 이유 (문맥 의존)**: "은행에서 예금을 찾았다"의 *은행*(금융기관)과 "강가에 있는 은행나무"의 *은행*(나무)은 글자가 같지만 의미가 다릅니다. BOW·Word2Vec은 *은행*을 **항상 같은 숫자/벡터**로 보지만, BERT 임베딩은 **문장마다 다른 벡터**를 만들어 의미를 구분합니다.

> 🟩 **입문자용 정리**: BOW·TF-IDF는 "단어를 센다"에 가깝고(전통 머신러닝과 잘 어울림), Word2Vec 이후는 "의미를 담는다"에 가깝습니다(딥러닝과 잘 어울림). 처음에는 **TF-IDF + 전통 분류기**로 시작해 감을 잡는 것을 권합니다(§5에서 실습).

### 4.3 🟩 BOW를 직접 만들어 보기 (실행 결과 O)

강의의 BOW 예시("커피 맛 최고 커피 향")를 scikit-learn으로 그대로 재현해 봅니다.

```python
# 🟩 보충 실습 — 설치: pip install scikit-learn
from sklearn.feature_extraction.text import CountVectorizer

docs = ["커피 맛 최고 커피 향"]

# 기본 설정은 한 글자 단어(맛, 향)를 버리므로, token_pattern을 조정해 1글자도 포함
vec = CountVectorizer(token_pattern=r"(?u)\b\w+\b")
X = vec.fit_transform(docs)

print("단어 사전:", vec.get_feature_names_out())
print("BOW 벡터:", X.toarray())
print("벡터 shape:", X.shape)
```

실제 실행 출력:

```text
단어 사전: ['맛' '최고' '커피' '향']
BOW 벡터: [[1 1 2 1]]
벡터 shape: (1, 4)
```

> 🟩 **해석**: 사전에는 서로 다른 단어 4종류(`맛, 최고, 커피, 향`)가 들어 있고, 벡터의 각 숫자는 **그 단어가 몇 번 나왔는지**를 뜻합니다. *커피*가 2번이라 `2`, 나머지는 1번이라 `1`입니다. shape `(1, 4)` 는 "문장 1개 × 단어 종류 4개"라는 의미입니다. 이렇게 BOW는 **순서를 버리고 횟수만** 남깁니다 — "커피 맛 최고"와 "최고 맛 커피"가 똑같은 벡터가 되는 게 BOW의 한계입니다.

## 5. 텍스트 분류에 쓰는 모델 🟦

### 5.1 두 갈래: 전통 머신러닝 vs 딥러닝 🟦

🟦 **전통적인 머신러닝 알고리즘** (BOW·TF-IDF와 함께 많이 사용)

- Naive Bayes, Logistic Regression, Decision Tree, Random Forest, SVM, XGBoost, LightGBM
- **특징**: 비교적 **적은 데이터에서도** 좋은 성능을 보임.

🟦 **딥러닝 모델** (텍스트의 순서·문맥을 학습)

- RNN, LSTM, GRU, CNN, Transformer, BERT, GPT
- 👉 각 모델의 구조와 특징은 **DAY1(자연어 처리를 위한 딥러닝 모델)** 에서 자세히 다뤘습니다. 이번 글에서는 "분류에 쓰인다"는 맥락만 짚고, 실습은 전통 머신러닝으로 진행합니다.

> 🟩 **언제 무엇을?**: 데이터가 적고 빠르게 결과가 필요하면 **TF-IDF + 전통 분류기**가 강력합니다. 문맥·뉘앙스가 중요하고 데이터가 충분하면 **딥러닝/BERT 계열**이 유리합니다. "딥러닝이 항상 낫다"는 단정은 피하세요 — 데이터 크기·과제·자원에 따라 다릅니다.

### 5.2 모델 학습 과정 🟦

강의가 제시한 학습 흐름입니다(딥러닝 기준).

```text
문장 → 벡터화 → 모델 입력 → 예측 → 손실 계산 → 역전파 → 가중치 업데이트 → 반복
```

> 🟩 전통 머신러닝(예: 나이브 베이즈)은 위 과정 중 "역전파·반복 학습"이 없고, 한 번의 `fit()` 으로 통계를 계산합니다. 아래 실습이 그 예입니다.

### 5.3 🟩 "TF-IDF + 나이브 베이즈"로 스팸 분류 (실행 결과 O)

전처리→벡터화→학습→평가의 전체 흐름을, **강의에 나온 개념(TF-IDF, 나이브 베이즈)을 scikit-learn 보충 코드로 구현**해 작은 한국어 예제에 적용해 봅니다. 데이터는 재현을 위해 직접 만든 소량의 합성 문장입니다.

```python
# 🟩 보충 실습 — 설치: pip install scikit-learn
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.naive_bayes import MultinomialNB
from sklearn.pipeline import Pipeline
from sklearn.model_selection import train_test_split
from sklearn.metrics import classification_report, confusion_matrix

# 1=스팸, 0=정상 (작은 합성 데이터)
spam = [
    "지금 회원가입하면 최신 노트북을 무료로 드립니다",
    "한정 이벤트 지금 클릭하면 경품 무료 증정",
    "당첨 되셨습니다 지금 링크를 눌러 확인하세요",
    "무료 쿠폰 지금 신청하고 혜택 받으세요",
    "광고 특가 할인 지금 구매하면 사은품 증정",
    "축하합니다 무료 상품권 당첨 지금 수령하세요",
    "긴급 계정 보안 확인 지금 링크 클릭 필요",
    "선착순 무료 체험 지금 신청하세요 혜택 가득",
    "대출 무이자 지금 신청 즉시 승인 가능",
    "무료 기프티콘 도착 지금 바로 확인하세요",
]
ham = [
    "오늘 회의는 오후 세시에 진행됩니다",
    "보내주신 자료 잘 받았습니다 감사합니다",
    "내일 점심 약속 시간 변경 가능할까요",
    "프로젝트 보고서 초안을 첨부해 드립니다",
    "주말에 가족과 영화를 보러 갔습니다",
    "회의록 정리해서 메일로 공유하겠습니다",
    "다음 주 일정 확인 부탁드립니다",
    "어제 보내주신 의견 반영해서 수정했습니다",
    "점심 메뉴는 김치찌개로 정했습니다",
    "발표 자료 준비 거의 끝나갑니다",
]
texts = spam + ham
labels = [1] * len(spam) + [0] * len(ham)

# 1) 먼저 훈련/평가로 분리 (데이터 누수 방지의 핵심)
X_train, X_test, y_train, y_test = train_test_split(
    texts, labels, test_size=0.3, random_state=42, stratify=labels)

# 2) TF-IDF와 분류기를 Pipeline으로 묶기
#    → Pipeline이 TF-IDF를 '훈련 데이터에만' fit 하므로 평가 정보가 새지 않음
clf = Pipeline([
    ("tfidf", TfidfVectorizer(token_pattern=r"(?u)\b\w+\b")),
    ("nb", MultinomialNB()),
])

clf.fit(X_train, y_train)            # 학습
pred = clf.predict(X_test)           # 평가셋 예측

print("훈련 문장 수:", len(X_train), "/ 평가 문장 수:", len(X_test))
print("예측:", list(int(p) for p in pred))
print("정답:", y_test)
print("\n혼동행렬 [행=실제, 열=예측]:")
print(confusion_matrix(y_test, pred))
print("\n분류 리포트:")
print(classification_report(y_test, pred, target_names=["정상(0)", "스팸(1)"], zero_division=0))

# 학습에 없던 새 문장 예측
new = ["지금 신청하면 무료 사은품을 드립니다", "내일 회의 자료 부탁드립니다"]
print("새 문장 예측(1=스팸, 0=정상):", list(int(p) for p in clf.predict(new)))
```

실제 실행 출력:

```text
훈련 문장 수: 14 / 평가 문장 수: 6
예측: [0, 1, 1, 1, 1, 0]
정답: [0, 1, 0, 1, 1, 0]

혼동행렬 [행=실제, 열=예측]:
[[2 1]
 [0 3]]

분류 리포트:
              precision    recall  f1-score   support

      정상(0)       1.00      0.67      0.80         3
      스팸(1)       0.75      1.00      0.86         3

    accuracy                           0.83         6
   macro avg       0.88      0.83      0.83         6
weighted avg       0.88      0.83      0.83         6

새 문장 예측(1=스팸, 0=정상): [1, 0]
```

> 🟩 **결과 해석 (숫자가 완벽하지 않은 게 정상)**:
> - **혼동행렬** `[[2 1],[0 3]]`: 실제 정상 3개 중 2개는 정상으로 맞혔지만 1개를 스팸으로 잘못 봤고(거짓 양성), 실제 스팸 3개는 모두 맞혔습니다.
> - **accuracy 0.83**: 6개 중 5개를 맞혔습니다. 데이터가 14개뿐이라 한 개만 틀려도 점수가 크게 흔들립니다 — **소량 데이터의 한계**를 그대로 보여 줍니다.
> - 새 문장 `"무료 사은품…"→스팸(1)`, `"회의 자료…"→정상(0)` 으로 잘 분류했습니다.
> - **왜 Pipeline인가**: TF-IDF의 단어 통계를 **훈련 데이터에만** 학습시키기 위해서입니다. 만약 전체 데이터로 먼저 `fit` 하면 평가 문장의 정보가 학습에 새어 들어가(**데이터 누수, data leakage**) 성능이 실제보다 부풀려집니다.

> 🟩 **주의**: 이 결과는 위 작은 합성 데이터·`random_state=42` 기준이라 데이터를 바꾸면 숫자도 달라집니다. **실제 한국어 분류**에서는 ① 형태소 분석기로 토큰을 정리하고, ② 훨씬 더 많은 데이터를 모으며, ③ 별도의 검증셋(validation set)이나 교차검증(cross-validation)으로 성능을 확인해야 합니다.

## 6. 성능 평가 지표 🟦

분류 모델은 다음 지표로 평가합니다.

| 지표 | 의미 |
|---|---|
| **Accuracy**(정확도) | 전체 예측 중 맞춘 비율 |
| **Precision**(정밀도) | 특정 클래스로 **예측한 것 중** 실제로 맞은 비율 |
| **Recall**(재현율) | **실제 해당 클래스 중** 모델이 찾아낸 비율 |
| **F1-Score** | Precision과 Recall의 **조화평균** |
| **Confusion Matrix** | 예측 결과를 표로 분석(맞춤/틀림의 종류) |
| **ROC-AUC** | 분류 모델의 전반적 성능을 하나의 값으로 |

> 🟩 **불균형 데이터에서는 정확도만 보면 안 됩니다**: 강의도 한계로 "데이터 불균형이 심하면 특정 클래스에 편향될 수 있다"고 지적합니다. 예를 들어 스팸이 전체의 10%뿐이라면, **"전부 정상"** 이라고만 찍어도 정확도가 90%입니다. 하지만 스팸은 하나도 못 잡았으므로 쓸모가 없습니다. 그래서 §5.3처럼 **precision·recall·F1** 을 함께 봐야 합니다.

> 🟩 **Precision vs Recall, 무엇이 중요?**: 스팸 필터에서 **정밀도(Precision)** 가 낮으면 정상 메일을 스팸으로 오분류해 중요한 메일을 놓치고, **재현율(Recall)** 이 낮으면 스팸이 받은편지함에 새어 듭니다. 어느 쪽 실수가 더 비싼지에 따라 중점 지표가 달라집니다.

> 🟩 **ROC-AUC는 언제 보나?**: ROC-AUC는 모델이 낸 **확률 또는 점수**를 기준으로 **임계값(threshold)을 바꿔 가며** 성능을 보는 지표입니다. 이진 분류에서 자주 쓰이고, 다중 분류에서는 `macro`·`weighted`·`one-vs-rest` 같은 계산 방식을 추가로 골라야 합니다. 그래서 입문 단계에서는 먼저 **혼동행렬·precision·recall·F1** 을 해석한 뒤, ROC-AUC를 보조 지표로 이해하는 것이 좋습니다.

### 6.1 실제 활용 분야와 한계 🟦

🟦 **활용 분야**: 전자상거래(상품 리뷰 분류), 금융(문서 자동 분류), 의료(진료 기록 분석), 법률(계약서 분류), 교육(자료 분류), SNS(감성 분석), 고객센터(문의 유형 분류), 이메일(스팸 탐지), 언론(뉴스 분류).

🟦 **장점**: 대량의 문서를 빠르게 자동 분류할 수 있고, 검색·추천·분석 시스템과 쉽게 연계됩니다.

🟦 **한계**: 학습 데이터 **품질에 크게 좌우**되며, **신조어·은어·오탈자·문맥 이해**가 어려운 경우 성능이 떨어집니다. 데이터 불균형이 심하면 특정 클래스로 편향될 수 있습니다.

## 7. 최신 텍스트 분류 기술 🟦

최근에는 BOW·TF-IDF로 단어 빈도만 보던 방식에서 벗어나, **대규모 텍스트를 미리 학습한 사전학습 언어모델(PLM)** 과 **생성형 대규모 언어모델(LLM)** 을 활용하는 방향으로 발전하고 있습니다. 이 모델들은 단어 하나의 의미를 넘어 **문장 전체의 문맥과 의도**까지 고려합니다.

### 7.1 BERT 계열 기반 분류 🟦

- **BERT**: Transformer의 **Encoder** 구조 기반 사전학습 모델. 문장의 **앞뒤 문맥을 동시에 고려**(양방향)합니다.
- Word2Vec·FastText는 *은행* 을 항상 같은 벡터로 보지만, BERT는 "통장을 만든 은행(금융)"과 "공원의 은행나무(나무)"를 **다른 벡터**로 구분합니다.
- **사전학습(Pre-training) → 미세조정(Fine-tuning)** 방식이라, 특정 업무 데이터가 적어도 좋은 성능을 낼 수 있습니다(단, 데이터·도메인 차이에 따라 결과는 달라집니다).
- 🟦 대표 모델: BERT, RoBERTa, DistilBERT, ALBERT, ELECTRA, DeBERTa / **한국어**: KoBERT, KR-BERT, KoELECTRA.

> 🟩 **용어 한 줄 정리**: **사전학습(Pre-training)** 은 대량의 일반 텍스트로 언어 자체를 먼저 익히는 단계, **미세조정(Fine-tuning)** 은 그 모델을 내 분류 과제 데이터로 조금 더 학습시키는 단계입니다.

### 7.2 생성형 LLM을 이용한 분류 🟦

LLM은 인터넷의 방대한 문서를 학습해 언어를 이해하므로, **별도 분류 모델을 새로 학습하지 않아도** 자연어 명령(Prompt)만으로 분류할 수 있습니다.

🟦 예: *"다음 문장을 긍정/부정 중 하나로 분류하시오. 문장: '배송이 예상보다 빨라 매우 만족스럽습니다.'"* → 모델: **긍정**

| 방식 | 설명 | 특징 |
|---|---|---|
| **Zero-shot** | 예시 없이 라벨 후보만 주고 분류 | 라벨링 데이터 불필요, 새 분야에 즉시 적용 |
| **Few-shot** | **몇 개의 예시**를 준 뒤 새 문장 분류 | 새 업무에 빠르게 적응, 실무에서 자주 사용 |
| **In-context Learning** | 모델 **가중치를 바꾸지 않고**, 프롬프트의 예시만으로 수행 | 재학습 없이 패턴 학습, 대규모 모델의 핵심 특징 |

> 🟩 **포인트**: Zero/Few-shot, In-context Learning은 모두 **"모델을 다시 훈련시키지 않는다"** 는 공통점이 있습니다. 학습 대신 **프롬프트(지시문)** 로 작업을 정의한다는 점이 전통적 분류와의 가장 큰 차이입니다.

### 7.3 RAG를 이용한 분류 🟦

**RAG(Retrieval-Augmented Generation, 검색 증강 생성)** 는 **검색(Retrieval) + 생성(Generation)** 을 결합한 기술입니다. LLM은 학습 시점 이후의 정보를 모르지만, RAG는 **외부 문서·데이터베이스에서 관련 정보를 검색**한 뒤 그 결과를 바탕으로 분류·답변합니다.

```text
사용자 질문
  ↓ Vector Database 검색
  ↓ 관련 문서 검색
  ↓ LLM 입력
  문서 분류 결과 출력
```

> 🟩 **검색 전에 준비가 필요합니다**: 위 흐름이 작동하려면, 그 전에 **문서 수집 → 텍스트 임베딩 생성 → Vector Database 저장**이라는 준비 단계가 먼저 끝나 있어야 합니다(§7.4의 전체 그림 참고). 즉 미리 문서를 벡터로 저장해 두어야, 질문이 들어왔을 때 비슷한 문서를 빠르게 찾을 수 있습니다.

🟦 대표 **Vector Database**: Qdrant, FAISS, Milvus, Chroma, Pinecone, Weaviate. (제품·설치·클라우드 API는 시점에 따라 달라지므로, 실제 사용 시 각 공식 문서를 확인하세요.)
🟦 RAG를 쓰면 **최신 문서·사내 데이터** 등 모델이 학습하지 않은 정보도 활용해 정확도를 높일 수 있습니다(사내 문서 분류, 상담 이력 분류, 법률·의료·연구 논문 분류 등).

### 7.4 하이브리드 모델과 최신 동향 🟦

실무에서는 한 가지 기법만 쓰기보다 **여러 기법을 결합한 하이브리드(Hybrid)** 사례가 늘고 있습니다. **통계 기반 특징(TF-IDF)** 과 **딥러닝 기반 의미 정보(임베딩)** 를 함께 써 성능을 끌어올리는 것이 목적입니다.

🟦 대표 조합(입문 단계에서는 **이름만 훑어도 충분**합니다): `TF-IDF + SVM`, `TF-IDF + XGBoost`, `BERT + Softmax`, `Sentence-BERT + Random Forest`, `Sentence-BERT + XGBoost`, `Embedding + Vector Search + LLM`, `RAG + LLM`, `GNN + Transformer`.

🟦 최근 기업은 단순 분류를 넘어 **검색·분류·요약·질의응답을 하나의 시스템으로 통합**하는 방향으로 발전하고 있습니다.

```text
문서 수집 → 텍스트 전처리 → 문서 임베딩 생성 → Vector DB 저장
  → 사용자 질의 → 유사 문서 검색(Retrieval) → LLM 분석
  → 문서 분류 → 요약 생성 → 질의응답 제공
```

### 7.5 기술 비교표 🟦

| 구분 | 기존 머신러닝 | BERT 계열 | 생성형 LLM | RAG 기반 | 하이브리드 |
|---|---|---|---|---|---|
| 문맥 이해 | 거의 불가능 | 매우 우수 | 매우 우수 | 매우 우수 | 매우 우수 |
| 추가 학습 | 필요 | 미세조정 필요 | 경우에 따라 불필요(Zero/Few-shot) | 검색 데이터 구축 | 구성에 따라 다름 |
| 최신 정보 반영 | 어려움 | 어려움 | 제한적 | 매우 우수 | 우수 |
| 학습데이터 요구량 | 많음 | 비교적 적음 | 매우 적거나 불필요 | 검색 문서 중심 | 중간 |
| 추론 속도 | 매우 빠름 | 빠름 | 상대적으로 느림 | 검색+추론으로 느림 | 중간 |
| 대표 활용 | 스팸·감성 분석 | 문서·감성·의도 분류 | 범용 분류, AI 비서 | 기업 문서 분류, 지식 검색 | 대규모 실무 AI |

> 🟩 **최신 모델도 조건이 중요합니다 (과장 금지)**: 위 비교표(강의 제공)의 "매우 우수 / 빠름" 등은 **일반적 경향**이지 절대 규칙이 아닙니다. BERT·생성형 LLM·RAG는 문맥을 잘 활용해 강력하지만, **항상** 전통적인 TF-IDF 모델보다 좋은 것은 아닙니다. 결과는 **라벨 데이터 품질, 도메인 차이, 검색 문서 품질, 추론 비용, 응답 속도**에 따라 달라집니다. 그래서 실무에서는 먼저 **`TF-IDF + Logistic Regression/SVM`** 같은 가벼운 기준 모델(baseline)을 만들어 비교한 뒤, 더 무거운 모델이 정말 필요한지 판단하는 것이 좋습니다.

### 7.6 향후 발전 방향 🟦

향후 텍스트 분류는 단순히 문서를 한 범주로 나누는 수준을 넘어, **의미를 종합적으로 이해·추론**하는 방향으로 발전하고 있습니다. 생성형 AI와 검색 기술이 결합되며 분류가 **검색·요약·추천·질의응답·정보추출**과 통합되고, **멀티모달 AI**(텍스트+이미지+표+음성)로 확장되고 있습니다. 기업은 LLM·RAG·벡터 DB·에이전트(Agent)를 결합한 **지능형 문서 처리 시스템**을 구축하는 사례가 늘고 있습니다.

## 8. 입문자가 자주 하는 실수 🟩

1. **벡터라이저를 전체 데이터로 학습**: `train_test_split` **전에** TF-IDF/사전을 만들면 데이터 누수. → 항상 훈련 데이터로만 `fit`(Pipeline 사용 권장).
2. **불균형 데이터에서 정확도만 보기**: 스팸 10% 문제에서 정확도 90%는 의미가 약함. → precision·recall·F1·혼동행렬 확인.
3. **다중 분류와 다중 라벨 혼동**: 다중 분류는 softmax(정답 1개), 다중 라벨은 라벨별 sigmoid(정답 여러 개). → 출력층·손실함수가 다름.
4. **한국어를 띄어쓰기로만 토큰화**: 조사·어미 때문에 같은 단어가 흩어짐. → 형태소 분석기 사용 고려.
5. **"최신 기술이 항상 최고"라는 가정**: 데이터·자원·속도 요구에 따라 전통 기법이 나을 때도 많음.
6. **작은 데이터의 점수를 과신**: 데이터가 적으면 점수가 크게 흔들림. → 더 많은 데이터·교차검증.

## 9. DAY2 핵심 정리

```text
텍스트 분류 = 텍스트(입력) → 미리 정해진 라벨(출력)
  과정: 수집 → 전처리 → 벡터화 → 학습 → 평가

종류
  - 이진 / 다중(softmax, 정답 1개) / 다중 라벨(라벨별 sigmoid, 정답 여러 개)

텍스트 → 숫자
  - 전처리: 형태소 분석 · 불용어 제거 · 토큰화 · 정수 인코딩 · 패딩
  - 벡터화: BOW · TF-IDF(횟수) → Word2Vec/FastText/GloVe(의미) → BERT(문맥)

모델
  - 전통 ML(NB, LogReg, SVM, XGBoost…) + BOW/TF-IDF: 적은 데이터에 강함
  - 딥러닝(RNN~Transformer/BERT/GPT): 문맥·뉘앙스에 강함 (DAY1 참고)

평가
  - accuracy 외 precision·recall·F1·confusion·ROC-AUC
  - 불균형 데이터는 정확도만 보면 안 됨

최신 기술
  - BERT 계열(Fine-tuning) / 생성형 LLM(Zero·Few-shot, In-context)
  - RAG(검색+생성, Vector DB) / 하이브리드(통계+딥러닝)
  - 최신 모델도 데이터·도메인·비용 조건에 따라 다름 → baseline과 비교
```
