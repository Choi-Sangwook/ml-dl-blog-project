# ML/DL 블로그 작성 단계별 프롬프트

아래 프롬프트에서 `[DAY]`, `[분야]`, `[주제]`, `[파일명]` 부분만 실제 작업에 맞게 바꿔 사용합니다.

```text
[DAY]  예: DAY1
[분야] 예: 머신러닝 또는 딥러닝
[주제] 예: 딥러닝 개념과 프레임워크 설치
```

---

## 1단계: Claude Code 초안 작성

사용 시점:

- PDF를 처음 정리할 때
- 글의 구성과 설명 흐름을 만들 때
- 결과를 `drafts/`에 저장할 때

```text
$write-ml-blog-post를 사용해 다음 작업을 진행해줘.

대상 PDF:
sources/[PDF 파일명]

목표:
파이썬 기본 문법을 알고 있지만 [분야]를 처음 배우는 독자를 위한
Velog 포스트의 기획과 전체 초안을 작성해줘.

작업 기준:
1. PDF 전체 페이지를 확인하고 페이지별 핵심 주제 지도를 먼저 만들어줘.
2. 텍스트가 거의 없는 페이지도 건너뛰지 말고 코드, 표, 수식, 그림이 있는지 확인해줘.
3. PDF의 핵심 내용을 빠뜨리지 않되, 원문 순서를 그대로 옮기지 말고 입문자가 이해하기 좋은 순서로 재구성해줘.
4. PDF에 있는 내용과 이해를 돕기 위해 추가한 설명을 구분해줘.
5. 주요 개념은 왜 필요한지, 동작 원리, Python 예제, 결과 해석, 주의사항 순서로 설명해줘.
6. 코드는 하나의 프레임워크 안에서 일관되게 작성하고 복사하여 실행할 수 있도록 구성해줘.
7. 딥러닝 코드라면 tensor shape, dtype, device, 출력층, 라벨 형식, 손실함수 조합을 확인해줘.
8. 실행하지 않은 결과값을 임의로 만들지 마.
9. PDF에서 확인할 수 없는 다음 DAY 주제는 추측하지 마.
10. 기존 파일은 수정하지 마.

출력 파일:
drafts/[DAY]_[분야]_[주제]_초안_v1.md

완료 후 PDF에서 다룬 주요 항목과 저장 경로를 알려줘.
```

## 2단계: Codex PDF 대조 검토

사용 시점:

- Claude Code가 초안을 작성한 뒤
- PDF 누락, 오류, 코드 문제를 검토할 때
- 결과를 `reviews/`에 저장할 때

```text
$write-ml-blog-post를 사용해 PDF와 최신 초안을 대조 검토해줘.

기준 PDF:
sources/[PDF 파일명]

검토할 초안:
drafts/[초안 파일명]

대상 독자:
파이썬 기본을 알고 있지만 [분야]를 처음 배우는 사람

검토 항목:
1. PDF 목차와 전체 페이지를 기준으로 빠진 내용을 찾아줘.
2. 텍스트 추출이 적은 페이지의 코드, 표, 수식, 그림도 직접 확인해줘.
3. 틀렸거나 과장되거나 오해하기 쉬운 설명을 찾아줘.
4. 입문자에게 설명이 부족한 부분과 지나치게 깊거나 불필요한 부분을 구분해줘.
5. 코드의 import, 변수 정의, 전처리, 실행 순서, 출력 해석을 확인해줘.
6. 데이터 누수와 train/validation/test 사용 방법을 확인해줘.
7. 문제 유형에 적절한 평가 지표를 사용했는지 확인해줘.
8. 딥러닝이라면 프레임워크 혼용, tensor shape, dtype, device,
   출력층, 라벨 형식, 활성화 함수와 손실함수 조합을 확인해줘.
9. 설치 명령이나 버전에 민감한 API는 현재 공식 문서 확인이 필요한지 표시해줘.
10. PDF에 없는 추가 설명이 PDF 원문처럼 표현되지 않았는지 확인해줘.
11. 원본 PDF와 초안은 수정하지 마.

검토 문서 구성:
- 전체 평가
- 높은 우선순위의 오류와 수정사항
- PDF 기준 누락 내용
- 더 자세히 설명할 내용
- 유용한 추가 내용
- 줄이거나 제거할 내용
- 바로 붙여 넣을 수 있는 수정 블록
- 우선순위 표
- 최종 권고

출력 파일:
reviews/[DAY]_[분야]_[주제]_Codex_통합검토안.md

여러 검토 파일로 나누지 말고 하나의 통합 검토안으로 작성해줘.
```

## 3단계: Claude Code 최종본 작성

사용 시점:

- Codex 통합 검토안이 만들어진 뒤
- 초안에 검토 내용을 반영할 때
- 결과를 `posts/`에 저장할 때

```text
$write-ml-blog-post를 사용해 최종 포스트를 작성해줘.

기준 PDF:
sources/[PDF 파일명]

기존 초안:
drafts/[초안 파일명]

Codex 통합 검토안:
reviews/[검토안 파일명]

작업 목표:
초안의 자연스러운 설명 흐름을 유지하면서 Codex 검토안의 정확성,
코드, 누락 내용과 입문자 설명 보강 사항을 반영한 최종본을 작성해줘.

반영 원칙:
1. PDF의 확인 가능한 내용과 실제 실행 가능한 동작을 가장 우선해줘.
2. Codex 검토안의 높은 우선순위 수정사항은 반드시 검토하고 반영해줘.
3. 중간·낮은 우선순위 제안은 입문 범위에 도움이 되는 것만 선택해줘.
4. PDF에 없는 추가 설명은 PDF 원문처럼 표현하지 마.
5. 초안과 검토안이 충돌하면 정확성 및 실행 검증 내용을 우선해줘.
6. 코드 예제의 프레임워크, 데이터 전처리와 평가 흐름을 일관되게 유지해줘.
7. 같은 설명, 비교표와 요약을 반복하지 마.
8. 다음 DAY 주제가 확인되지 않았다면 임의로 예고하지 마.
9. PDF, 초안, 검토안은 수정하지 마.

출력 파일:
posts/[DAY]_[분야]_[주제]_최종본.md

완료 후 다음 내용을 알려줘:
- 반영한 주요 수정사항
- 반영하지 않은 선택적 제안과 이유
- 실행하거나 확인하지 못한 부분
- 최종 저장 경로
```

## 4단계: Codex 최종 품질 점검

권장 선택 단계입니다.

사용 시점:

- Velog에 게시하기 직전
- 검토사항이 실제 최종본에 반영되었는지 확인할 때
- 원칙적으로 파일을 수정하지 않고 결과만 보고받을 때

```text
$write-ml-blog-post를 사용해 게시 전 최종 품질 점검을 해줘.

기준 PDF:
sources/[PDF 파일명]

기존 검토안:
reviews/[검토안 파일명]

최종 포스트:
posts/[최종본 파일명]

점검 목표:
PDF의 핵심 내용과 Codex 통합 검토안이 최종본에 올바르게 반영되었는지 확인해줘.

점검 항목:
1. 높은 우선순위 수정사항의 반영 여부
2. PDF 핵심 내용의 최종 누락 여부
3. 기술적 오류나 과도한 단정 표현
4. 코드 블록의 실행 가능성과 앞뒤 설명 일치 여부
5. 데이터 누수와 평가 방식
6. 딥러닝의 tensor shape, dtype, device, 출력층과 손실함수 조합
7. Markdown 제목, 표, 링크, 코드 fence와 UTF-8 상태
8. 중복 설명과 불필요하게 긴 부분

이번 단계에서는 최종 포스트를 직접 수정하지 마.

결과는 다음 세 구역으로 간단히 보고해줘:
- 게시 가능
- 게시 전 반드시 수정
- 선택적으로 개선

문제가 없다면 명확히 "게시 가능"이라고 알려줘.
```

## 5단계: 최종 점검에서 문제가 발견된 경우

Claude Code에 다시 수정을 맡길 때 사용합니다.

먼저 Codex의 최종 점검 결과를 다음 파일로 저장합니다.

```text
reviews/[DAY]_[분야]_[주제]_Codex_최종점검.md
```

Claude Code 요청:

```text
$write-ml-blog-post를 사용해 최종 포스트의 수정본을 작성해줘.

현재 최종본:
posts/[현재 최종본 파일명]

Codex 최종 점검:
reviews/[최종점검 파일명]

작업 기준:
1. "게시 전 반드시 수정" 항목을 우선 반영해줘.
2. 선택적 개선은 글의 명확성과 입문 범위를 해치지 않는 경우에만 반영해줘.
3. 기존 최종본을 덮어쓰지 마.
4. 변경한 부분과 반영하지 않은 항목을 알려줘.

출력 파일:
posts/[DAY]_[분야]_[주제]_최종본_v2.md
```

## 기존 완성본을 새 기준으로 간단히 확인할 때

```text
$write-ml-blog-post를 사용해 기존 완성본을 간단히 점검해줘.

기준 PDF:
sources/[PDF 파일명]

기존 완성본:
posts/[완성본 파일명]

처음부터 새로 작성하지 말고 다음 항목만 확인해줘:
- PDF 핵심 내용 누락
- 명백한 기술 오류
- 실행되지 않는 코드
- 데이터 누수
- 문제 유형에 맞지 않는 평가
- 과도한 단정 표현
- Markdown 오류

기존 완성본을 수정하지 말고, 재검토가 필요한 경우에만 이유와 수정 위치를 알려줘.
```

## 여러 PDF를 하나의 글로 정리할 때

```text
$write-ml-blog-post를 사용해 다음 PDF들을 하나의 글로 정리해줘.

대상 PDF:
- sources/[PDF 1]
- sources/[PDF 2]
- sources/[PDF 3]

작업 기준:
1. 각 PDF의 페이지별 주제를 먼저 정리해줘.
2. 중복되는 내용은 한 번만 설명해줘.
3. PDF별 출처와 주제 경계를 내부적으로 추적해줘.
4. 서로 충돌하는 설명이 있으면 임의로 합치지 말고 차이를 알려줘.
5. 한 글이 지나치게 길어지면 DAY를 나누는 방안을 먼저 제안해줘.
6. 사용자가 분할을 요청하지 않았다면 임의로 여러 파일을 만들지 마.
```

## 빠른 운영 요약

```text
Claude Code:
sources PDF → drafts 초안

Codex:
sources PDF + drafts 초안 → reviews 통합 검토안

Claude Code:
sources PDF + drafts 초안 + reviews 검토안 → posts 최종본

Codex 선택 점검:
sources PDF + reviews 검토안 + posts 최종본 → 게시 가능 여부
```
